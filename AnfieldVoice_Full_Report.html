Solution Overview

AnfieldVoice replaces the estate intercom, visitor logbook, delivery register, and resident phone directory with a single, integrated platform. Every arrival, every call, every access decision is recorded, searchable, and attributable.

How It Works

1. Resident Authorises a Visit

Using the AnfieldVoice mobile app, a resident creates a Visitor Ticket. The system generates a one-time PIN valid for today's date only. The resident shares this PIN with their visitor.

2. Visitor Arrives at the Gate

The visitor enters their PIN at the weatherproof touchscreen terminal. The system instantly retrieves their ticket and displays it on the Security Dashboard.

3. Security Verifies and Admits

The security officer sees the ticket details — visitor name, vehicle, resident who authorised them — and visually confirms. If everything matches, admission takes seconds.

4. Audit Trail Created

Every step is logged immutably: ticket creation, PIN validation, admission decision, officer identity, and any exceptions. Available to administrators and never expires.

Visitor Management Workflows

1. Standard Visitor (One-Time)

Resident creates ticket in app → System generates one-day PIN → Visitor enters PIN at gate terminal → Security verifies and admits → Everything logged.

Total time: ~20-30 seconds from PIN entry to admission.

2. Recurring Visitor (Daily — Domestic Worker, School Transport)

Resident taps saved profile → "Generate Today's PIN" (5 seconds) → Visitor enters PIN → Dashboard shows "Verified Profile" badge → Security quick admit.

Total time: ~10-15 seconds. No phone call needed. Profile created once, used every day.

3. Unexpected Visitor (No PIN)

Visitor has no PIN → Terminal calls resident via mobile app → Resident answers and issues a PIN → Standard admit. No answer? Visitor is not admitted.

Event is logged as "Unexpected — No PIN — No Resident Response".

4. Delivery Management

Driver arrives at gate → Dashboard shows resident's pre-approval (Accept / Hold at Gate / Redirect) → Delivery logged with driver details, time, resolution.

No more "we have a package for apartment X" phone calls. Every delivery is accounted for.

Security Architecture — Six Layers of Verification

AnfieldVoice is designed on a defence-in-depth model. No single component failure or compromise can grant unauthorised access to the estate or resident data.

Layer 1 — Resident Authorisation

Every visit begins with an explicit resident action: creating a ticket and generating a PIN. No visitor can be processed without prior resident authorisation.

Layer 2 — Time-Bound PIN

PINs are valid for today's date only. Expired PINs are rejected automatically. The resident sets the validity window (default: 24 hours). Yesterday's PIN will not work today.

Layer 3 — Ticket Retrieval

When a PIN is entered at the gate, the system retrieves exactly one ticket — for one visitor, on one day. If the ticket does not exist or has been revoked, access is denied.

Layer 4 — Security Officer Verification

The security officer reviews the ticket details — visitor name, vehicle, category — and visually confirms the visitor before admission. The officer's identity is recorded.

Layer 5 — Exception Handling

If any detail does not match (wrong vehicle, different person, unexpected time), the officer can call the resident directly from the dashboard. No admission without resolution.

Layer 6 — Complete Audit Trail

Every decision is logged immutably: who created the ticket, when the PIN was used, which officer processed the visitor, the admission decision, and any exceptions.

Design Principles

Non-resident PA never appears as occupant
A letting agent managing 20 apartments should never receive gate calls or generate PINs for those units. They manage the tenancy — they don't live there.

Roles are additive, not exclusive
A resident who owns their unit can be both Resident and Property Admin. A security officer can also be Maintenance. No forced choices.
